Главная Подключение Сигналы CAN фреймы FreeRTOS HOWTO

CAN → WiFi Bridge

ESP32-S3 TWAI + TJA1050 → WiFi → SavvyCAN  |  ESP-IDF v5.3.1  |  v2.0 TWAI build 420

Обзор проекта

Устройство читает CAN-шину автомобиля через встроенный контроллер TWAI (Two-Wire Automotive Interface) ESP32-S3 и внешний трансивер TJA1050, затем передаёт фреймы по WiFi на любой клиент, поддерживающий протокол GVRET: компьютер с SavvyCAN, телефон с Telnet-клиентом, или любой TCP-клиент.

Связь двусторонняя: клиент может не только получать CAN-фреймы, но и отправлять их в шину, а также менять параметры CAN на лету.

Преимущества TWAI: Встроенный аппаратный CAN контроллер ESP32-S3. Всего 2 провода (TX/RX) вместо 4 (SPI). Аппаратный FIFO из 32 фреймов. Нет зависимости от внешнего контроллера.

Маршрутизация CAN-фреймов

CAN-фрейм получен из TWAI FIFO (twai_receive)
        │
        ├── isWifiActive == true  (TCP клиент подключён и прислал данные)
        │       └─▸ wifiGVRET.sendFrameToBuffer()
        │               └─▸ отправляется TCP клиенту при flush (каждые 20 мс)
        │
        └── isWifiActive == false (никто не подключён)
                └─▸ serialGVRET.sendFrameToBuffer()
                        └─▸ буфер очищается без отправки
isWifiActive становится true когда TCP-клиент первый раз присылает любые данные. SavvyCAN делает это автоматически при подключении (отправляет 0xE7).

SavvyCAN автоматически: отправит 0xE7 (binary mode), синхронизирует время, запросит параметры шин и начнёт получать фреймы.

Двусторонняя связь (GVRET команды)

GVRET (Generalized Vehicle Reverse Engineering Tool) — бинарный протокол для передачи CAN-фреймов и управления устройством. Все команды начинаются с байта 0xF1.
КомандаКодНаправлениеОписание
0xE7—Host → ESP32Переключение в бинарный режим GVRET
BUILD_CAN_FRAMEF1 00Host → ESP32Отправить CAN-фрейм в шину через TWAI
CAN_FRAME_DATAF1 00ESP32 → HostПринятый CAN-фрейм
TIME_SYNCF1 01Host ↔ ESP32Синхронизация времени (мкс)
SETUP_CANBUSF1 05Host → ESP32Изменить скорость / режим CAN
GET_CANBUS_PARAMSF1 06Host → ESP32Запрос текущих параметров CAN
GET_DEV_INFOF1 07Host → ESP32Версия прошивки и номер билда
KEEPALIVEF1 09Host → ESP32Проверка связи (ответ: F1 09 DE AD)
ECHO_CAN_FRAMEF1 0BHost → ESP32Зеркалирование фрейма
GET_NUMBUSESF1 0CHost → ESP32Количество CAN шин (ответ: 1)
GET_EXT_BUSESF1 0DHost → ESP32Параметры дополнительных шин
SET_EXT_BUSESF1 0EHost → ESP32Настроить дополнительные шины
Listen Only режим: по умолчанию CAN0ListenOnly = false — ESP32 может передавать в шину. Для безопасного мониторинга включите Listen Only через команду F1 05 или в config_idf.h.

TWAI RX FIFO = 32 фрейма. При 3800 фр/с это ~8 мс запаса. Главный цикл работает с тиком 1 мс — FIFO практически не заполняется. Пропускная способность WiFi (~5200 фр/с через TCP) тоже с запасом.

Исправленные ошибки

Bug #1 — Crash при старте: отсутствовал nvs_flash_init()

Файл: main/main.cpp

WiFi-стек ESP-IDF хранит RF-калибровки в NVS. Без nvs_flash_init() вызов esp_wifi_init() завершался паникой ядра (guru meditation error).

// Добавлено перед wifi_manager_setup():
esp_err_t nvs_ret = nvs_flash_init();
if (nvs_ret == ESP_ERR_NVS_NO_FREE_PAGES ||
    nvs_ret == ESP_ERR_NVS_NEW_VERSION_FOUND) {
    nvs_flash_erase();
    nvs_flash_init();
}

Bug #2 — Зависание главного цикла: блокирующий accept()

Файл: main/wifi_manager_idf.cpp

TCP listen-сокет был в блокирующем режиме. accept() блокировал весь поток — CAN-фреймы переставали читаться.

// Добавлено после открытия listen_fd:
fcntl(listen_fd, F_SETFL, fcntl(listen_fd, F_GETFL, 0) | O_NONBLOCK);

Bug #3 — Неверные метки времени: micros() делил на 1000

Файл: main/gvret_comm.cpp

esp_timer_get_time() уже возвращает микросекунды. Лишнее деление давало миллисекунды.

// Было:   #define micros() ((uint32_t)(esp_timer_get_time() / 1000))
// Стало:  #define micros() ((uint32_t)(esp_timer_get_time()))

Транспорт: TCP и UDP

TCP — порт 23

Все CAN-данные и управляющие команды. TCP гарантирует доставку и порядок байт — критично для бинарного GVRET.

Клиент → 192.168.4.1:23 (TCP)
  ↑↓ CAN-фреймы (GVRET)
  ↑↓ Команды управления

UDP — порт 17222

Только авто-обнаружение. Каждую секунду ESP32 broadcast отправляет 4 байта:

ESP32 → 255.255.255.255:17222
Payload: 1C EF AC ED

SavvyCAN использует это для автоматического поиска устройства.
