Всего 2 провода — никакого SPI, никакого внешнего CAN контроллера. TWAI — это встроенная периферия ESP32-S3, работающая напрямую через GPIO. TJA1050 — автономный трансивер (не на плате HW-184), только преобразование уровней.

TWAI — обмен ESP32 ↔ TJA1050

TWAI (Two-Wire Automotive Interface) — встроенный CAN контроллер ESP32-S3. Работает через два GPIO: TX и RX. Никакого протокольного оверхеда (как SPI) — TWAI периферия напрямую генерирует и принимает CAN битовый поток на скорости шины.

TX (ESP32 → TJA1050)

GPIOGPIO 4
Уровень HIGH3.3V (рецессивный)
Уровень LOW0V (доминантный)
ПриёмникTJA1050 TXD вход (VIH ~2.0V)
СогласованиеПрямое подключение, 3.3V > VIH

RX (TJA1050 → ESP32)

GPIOGPIO 5

После делителя0/3.33V (1к/2к)
ПриёмникESP32 GPIO 5 (макс. 3.6V)
СогласованиеРезисторный делитель 5V × 2k/(1k+2k) = 3.33V

Последовательность приёма CAN фрейма (TWAI)

1. CAN фрейм приходит на шину (CANH/CANL)
2. TJA1050 преобразует дифференциальный сигнал → TTL (5V)
3. Делитель 1к/2к снижает 5V → 3.33V
4. TWAI периферия ESP32 принимает битовый поток через GPIO 5
5. TWAI аппаратно проверяет CRC, выполняет ACK, декодирует фрейм
6. Фрейм помещается в RX очередь FreeRTOS (32 фрейма)
7. Задача вызывает twai_receive(&msg, pdMS_TO_TICKS(10))
8. Полученный twai_message_t содержит: ID, DLC, data[8], flags

TX: 3.3V → 5V вход TJA1050 — работает напрямую. TJA1050 имеет VIH = 2.0V (типичный CMOS порог). Сигнал 3.3V от ESP32 уверенно распознаётся как HIGH. Никаких преобразователей уровня не нужно.
RX: 5V → 3.3V (делитель обязателен!) — TJA1050 выдаёт 5V TTL на выходе RXD. ESP32-S3 GPIO допускает максимум 3.6V. Резисторный делитель 1к (верхний) + 2к (нижний) снижает 5V до 5 × 2/(1+2) = 3.33V — безопасно для ESP32.

GVRET протокол (WiFi ↔ SavvyCAN)

GVRET (Generalized Vehicle Reverse Engineering Tool) — бинарный протокол для передачи CAN фреймов по TCP. Используется SavvyCAN для связи с адаптерами. Передаётся по Telnet порту 23.

Исходящий фрейм (ESP32 → SavvyCAN)

Байт   Описание                  Пример
────────────────────────────────────────────
  0    0xF1 (заголовок GVRET)     F1
  1    0x00 (команда: CAN frame)   00
 2-5   timestamp (32-бит LE, мкс)  40 42 0F 00  (= 1000000 мкс = 1 сек)
 6-9   CAN ID (32-бит LE)          E0 07 00 00  (= 0x7E0)
       бит 31 = extended ID
 10    DLC | (bus << 4)            08  (8 байт, шина 0)
11-18  DATA (до 8 байт)            02 01 0C 00 00 00 00 00

Входящий фрейм (SavvyCAN → ESP32 для отправки в CAN)

Из GVRET state machine, команда BUILD_CAN_FRAME (0x00):

Байт   Описание
────────────────────────────────
 0-3   CAN ID (32-бит LE, бит 31 = extended)
  4    Номер шины (0 или 1)
  5    DLC (длина данных, 0-8)
 6+    DATA (DLC байт)

Инициализация связи

// Последовательность при подключении SavvyCAN:
1. SavvyCAN подключается к TCP:23
2. Отправляет 0xE7 → включает binary mode
3. Отправляет 0xF1 + 0x07 (GET_DEV_INFO) → ESP32 отвечает версией
4. Отправляет 0xF1 + 0x0C (GET_NUM_BUSES) → ответ: 1 шина
5. Отправляет 0xF1 + 0x06 (GET_CANBUS_PARAMS) → скорость, режим
6. Фреймы начинают приходить автоматически

TWAI RX FIFO32 фреймаFreeRTOS очередь (rx_queue_len=32). При переполнении — фрейм теряется и срабатывает алерт TWAI_ALERT_RX_QUEUE_FULL
Serial buffer1024 байтПромежуточный буфер GVRET данных
WiFi TX buffer2048 байтБуфер TCP отправки, flush каждые 20 мс
Потеря фреймов: при очень высокой загрузке шины (>3800 фр/с @500K) 32-фреймовый TWAI RX FIFO может переполниться. Однако это значительно лучше, чем внешние контроллеры с 2 буферами. При переполнении срабатывает TWAI_ALERT_RX_QUEUE_FULL. Решение: увеличить приоритет задачи чтения или использовать TWAI_MODE_LISTEN_ONLY для снижения нагрузки.
