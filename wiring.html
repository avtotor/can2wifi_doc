Путь сигнала (от машины до SavvyCAN)

0ПК + CANable 3.0 (имитация автомобиля) — Вместо реального автомобиля используется CANable 3.0 (STM32F072 с прошивкой candleLight). ПК с SavvyCAN подключается к CANable по USB (USB-A/USB-C) и отправляет тестовые CAN-фреймы. CANable преобразует их в дифференциальный сигнал CANH/CANL. Три выходных пина: GND, CAN_H, CAN_L. Перемычка JP1 замыкается для включения терминирующего резистора 120 Ом. Синий LED моргает при прохождении фреймов. GND CANable обязательно соединяется с общей землёй (GND ESP32 / TJA1050).
1ECU → CAN шина → TJA1050 — ECU автомобиля формирует CAN фрейм (SOF + ID + DLC + Data + CRC + ACK + EOF). Биты передаются дифференциально по витой паре CANH/CANL. TJA1050 преобразует дифференциальный сигнал (ΔV=2V) в одиночный TTL (0/5V) на линии RX. Время передачи фрейма: ~128 мкс @ 500K (64 бита данных + overhead).
2TJA1050 RX → делитель напряжения → ESP32 GPIO 5 (TWAI RX) — TJA1050 выдаёт 5V TTL на линии RX. Делитель 1к/2к снижает до 3.33V (безопасно для GPIO). Встроенный TWAI периферийный контроллер ESP32-S3 принимает битовый поток, выполняет bit stuffing decode, проверяет CRC-15, отправляет ACK, парсит ID/DLC/Data и помещает готовый фрейм в аппаратный RX FIFO на 32 фрейма. При переполнении — старые фреймы теряются, но 32 фрейма это значительно лучше, чем 2 буфера у MCP2515.
3canManager.loop() → twai_receive() — Каждую итерацию главного цикла (~1 мс) ESP32 вызывает canManager.loop() → ESP32CAN::available() проверяет наличие фреймов в TWAI FIFO. Если фрейм готов — ESP32CAN::read() вызывает twai_receive() (non-blocking, timeout=0), конвертирует twai_message_t в CAN_FRAME. Никакого SPI — прямой доступ к периферии через регистры. Файл: esp32_can_idf.cpp. Время: ~5 мкс.
4CAN_FRAME → GVRET serialize — canManager.displayFrame() вызывает wifiGVRET.sendFrameToBuffer(), которая сериализует фрейм в GVRET binary формат: 0xF1 (header) + 0x00 (cmd) + timestamp 4B + CAN ID 4B (бит 31 = extended) + (DLC | bus<<4) + data bytes. Результат записывается в WiFi TX буфер (2048 байт). Файл: gvret_comm.cpp. Время: ~5 мкс.
5WiFi TX buffer → TCP:23 → SavvyCAN — Каждые 20 мс (SER_BUFF_FLUSH_INTERVAL) или при заполнении буфера wifi_manager_send_buffered_data() отправляет накопленные GVRET пакеты через TCP сокет на порт 23. Данные проходят через lwIP стек (Core 0), WiFi стек формирует 802.11 фреймы, радио передаёт по каналу 1. SavvyCAN принимает TCP поток, декодирует GVRET binary, отображает фреймы в таблице. Файл: wifi_manager_idf.cpp.

Обратный путь (SavvyCAN → CAN шина)

ASavvyCAN → TCP:23 — Пользователь нажимает Send в SavvyCAN. Фрейм кодируется в GVRET binary (0xF1 + 0x00 + ID + bus + DLC + data) и отправляется по TCP.
BESP32 → GVRET decode — wifi_manager_loop() читает TCP данные (non-blocking), wifiGVRET.processIncomingByte() собирает фрейм через state machine (IDLE → GET_COMMAND → BUILD_CAN_FRAME).
CESP32 TWAI TX → TJA1050 → CAN шина — CANManager::sendFrame() → ESP32CAN::sendFrame() вызывает twai_transmit() с заданным таймаутом. TWAI контроллер выполняет арбитраж и передаёт фрейм через GPIO 4 (TX) в TJA1050. TX линия безопасна: ESP32 выдаёт 3.3V, TJA1050 воспринимает это как HIGH (VIH ≥ 2.0V для 5V логики).

Пропускная способность: один GVRET фрейм = ~19 байт. Буфер 2048 байт = ~107 фреймов. При flush каждые 20 мс = ~5350 фреймов/с. Типичный автомобиль генерирует 500–3000 фреймов/с — запас есть. TWAI FIFO на 32 фрейма позволяет пережить пиковые нагрузки без потерь (MCP2515 имел всего 2 буфера).

На модуле TJA1050 НЕТ стабилизатора! Вся логика работает от VCC напрямую (5V). Выход RX имеет уровень 5V. Поэтому на линии RX → GPIO 5 нужен делитель напряжения! Вход TX принимает 3.3V как HIGH (порог VIH ≥ 2.0V для 5V логики) — делитель не нужен.

Делитель напряжения (TJA1050 RX → GPIO 5)

TJA1050 RX (5V) ──[R1 = 1 кОм]──┬── GPIO 5 (3.33V)
                                 │
                             [R2 = 2 кОм]
                                 │
                               GND

Формула: Vout = 5V * 2k / (1k+2k) = 3.33V
Нет 2к? Два по 1к последовательно.

Без делителя GPIO 5 получит 5V → повреждение ESP32! Линия TX (GPIO 4 → TJA1050 TX) безопасна: ESP32 выдаёт 3.3V, TJA1050 воспринимает 3.3V как HIGH (порог VIH ≥ 2.0V для 5V логики). Делитель нужен ТОЛЬКО на RX.

Нет CNF регистров! В отличие от MCP2515, где нужно было вручную вычислять CNF1/CNF2/CNF3 под кварц 8 МГц, TWAI использует готовые макросы ESP-IDF. Тайминг рассчитан под внутренний APB clock (80 МГц) — никаких внешних кварцев.

Конфигурация TWAI (в коде)

// config_idf.h
#define TWAI_TX_PIN    4     // TWAI TX → TJA1050 TX (напрямую)
#define TWAI_RX_PIN    5     // TWAI RX ← TJA1050 RX (через делитель 1к/2к!)

// esp32_can_idf.cpp — инициализация TWAI драйвера
twai_general_config_t g_config = TWAI_GENERAL_CONFIG_DEFAULT(TX, RX, TWAI_MODE_NORMAL);
g_config.rx_queue_len = 32;      // 32 фрейма в FIFO (MCP2515 имел только 2)

twai_timing_config_t t_config = TWAI_TIMING_CONFIG_500KBITS();  // скорость CAN
twai_filter_config_t f_config = TWAI_FILTER_CONFIG_ACCEPT_ALL();  // принимать все фреймы

twai_driver_install(&g_config, &t_config, &f_config);  // установить драйвер
twai_start();  // запустить CAN контроллер

Терминация CAN (120 Ом)

Проверьте терминацию! Большинство standalone TJA1050 модулей НЕ имеют съёмной перемычки для терминации (в отличие от HW-184 с J3). Если на модуле есть впаянный резистор 120 Ом между CANH и CANL — его нужно удалить при подключении к автомобилю.
Если терминации нет на вашем модуле — просто подключайте к машине. В автомобиле CAN шина уже терминирована двумя резисторами по 120 Ом на концах.
Правильная терминация CAN:

 [120Ω]──────────шина──────────[120Ω]
  ECU1           ...              ECU2
                  │
            TJA1050 модуль
         (без терминации!)

Импеданс шины: 60 Ом (два по 120 параллельно)

Тестовый стенд (без автомобиля)

CANable 3.0 — USB-CAN адаптер на базе STM32F072 с прошивкой candleLight (gs_usb). Определяется в системе как SocketCAN (Linux) или через WinUSB драйвер (Windows). SavvyCAN поддерживает его напрямую через Connection → socketCAN / candleLight.

Узел120 Ом

Два узла с двумя терминаторами по 120 Ом — правильная терминация CAN шины (60 Ом).

GND обязателен! Без общей земли между CANable и ESP32/TJA1050 дифференциальный сигнал CAN не будет корректно считываться. GND CANable подключается к любой точке общей земли: GND пин TJA1050 или GND пин ESP32.

Если синий LED моргает — фреймы идут по CAN шине. Если не моргает — проверьте: 1) скорость CAN в SavvyCAN совпадает (500K), 2) CANH/CANL не перепутаны, 3) GND подключён, 4) терминация.
