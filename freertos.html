Архитектура реального времени, задачи, тайминг главного цикла

Архитектура приложения

Однопоточная архитектура: весь пользовательский код выполняется в одной задаче app_main на Core 1. Собственные задачи (xTaskCreate) не создаются. Системные задачи WiFi/lwIP работают на Core 0 автоматически.

Главный цикл (app_main)

Инициализация (выполняется один раз)

extern "C" void app_main(void) {
    uart_serial_init();           // UART0 115200 8N1
    app_init_gpio();              // LED GPIO2 как выход
    // ... логирование chip info ...
    nvs_flash_init();             // Non-Volatile Storage (настройки)
    load_settings();              // CAN 500K, WiFi AP, GVRET mode
    canManager.setup();           // TWAI driver install + start (GPIO 4 TX, 5 RX)
    vTaskDelay(pdMS_TO_TICKS(250));  // ждём стабилизации TWAI
    wifi_manager_setup();         // WiFi AP + TCP:23 + UDP:17222
    ...
}

Бесконечный цикл (polling)

uint32_t last_flush = (uint32_t)(esp_timer_get_time() / 1000);

for (;;) {                              // бесконечный цикл
    canManager.loop();                  // 1. TWAI receive → read CAN frames (32-frame FIFO)
                                         //    → displayFrame() → GVRET serialize

    wifi_manager_loop();                // 2. accept clients, poll TCP recv
                                         //    → processIncomingByte() (GVRET RX)
                                         //    → UDP beacon every 1000 ms

    // 3. WiFi TX: flush buffer if full or 20ms elapsed
    uint32_t now = (uint32_t)(esp_timer_get_time() / 1000);
    if (now - last_flush >= 20) {       // SER_BUFF_FLUSH_INTERVAL = 20 ms
        wifi_manager_send_buffered_data();
        last_flush = now;
    }

    // 4. Serial: read up to 128 bytes from UART
    for (int i = 0; i < 128; i++) {
        if (!uart_serial_read_byte(&b)) break;
        serialGVRET.processIncomingByte(b);
    }

    vTaskDelay(pdMS_TO_TICKS(1));       // 5. yield → 1ms → IDLE1 → watchdog
}

Тайминг одной итерации главного цикла

CPU загружен всего ~4%! Двухъядерный ESP32-S3 используется минимально. Core 0 занят системными задачами WiFi/lwIP. Core 1 большую часть времени простаивает в IDLE. TWAI работает аппаратно — минимальная нагрузка на CPU. Есть огромный запас для дополнительных задач.

Глобальные объекты (разделяемое состояние)

// config_idf.c — глобальные экземпляры

EEPROMSettings settings;          // CAN speed, WiFi SSID, режим и т.д.
SystemSettings SysSettings;       // numBuses, WiFi client socket

GVRET_Comm_Handler serialGVRET;  // GVRET для UART (serial console)
GVRET_Comm_Handler wifiGVRET;    // GVRET для WiFi/TCP (SavvyCAN)

CANManager canManager;            // управление CAN шиной и фреймами

// Буферы (static arrays)
uint8_t wifiTxBuffer[2048];       // WiFi TX буфер
uint8_t serialBuffer[1024];      // Serial TX буфер
Все объекты доступны из одного потока — race conditions невозможны. Но если добавить вторую задачу (например, CAN polling на отдельном ядре), потребуются мьютексы для canManager и буферов.

Стек main task = 3584 байт. Это мало! Глубокая рекурсия или большие локальные массивы вызовут stack overflow. Canary-проверка включена — при переполнении ESP32 покажет паник-лог. При добавлении функционала увеличьте CONFIG_ESP_MAIN_TASK_STACK_SIZE.

Системные задачи ESP-IDF (Core 0)

Preemptive scheduling: задача WiFi (prio 23) может прервать наш main (prio 1) в любой момент. Это означает, что при активной WiFi передаче обработка CAN может быть задержана на несколько мкс. На практике это не проблема — TWAI имеет аппаратный FIFO из 32 фреймов, что даёт запас ~4 мс при 8000 фр/с.

FreeRTOS в исходном коде

// Только 2 файла включают FreeRTOS:

// main.cpp
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

// esp32_can_idf.cpp
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"
#include "driver/twai.h"

// Используемые API:
vTaskDelay(pdMS_TO_TICKS(ms))    // задержка в миллисекундах
portTICK_PERIOD_MS               // = 1 (при HZ=1000)
esp_timer_get_time()             // микросекунды (не FreeRTOS, а ESP-IDF)
twai_receive(&msg, 0)            // неблокирующее чтение CAN фрейма из TWAI FIFO
twai_transmit(&msg, pdMS_TO_TICKS(10)) // отправка фрейма с таймаутом 10 мс

Текущая архитектура достаточна для 500K CAN @ ~3800 фр/с. С TWAI FIFO из 32 фреймов запас существенно больше, чем с внешним SPI контроллером. Оптимизация нужна только при: 1) CAN 1M @ >7000 фр/с, 2) множественных WiFi клиентах, 3) одновременном логировании на SD.
